# 평생곁에 — 기술 상세 문서 (실제 구현 기준)

> 이 문서는 **실제로 구현되어 동작하는 코드**(`backend/rag.py`, `backend/fds.py`, `backend/server.py`, `demo/index.html`)를 근거로,
> 각 기술이 **① 어떤 데이터를 가져와 → ② 어떻게 가공하고 → ③ 어떤 결과를 만드는지**를 단계별로 설명한다.
> "설계/계획"이 아니라 **현재 돌아가는 파이프라인**을 기술한다. (설계·확장 영역은 §8에서 정직하게 구분)

## 0. 전체 데이터 흐름 한 장

```
┌──────────────────────────── 입력 ────────────────────────────┐
│  (대화) 어르신/가족/창구의 질문        (거래) 이체·출금 이벤트   │
└───────────────┬──────────────────────────────┬───────────────┘
                │                              │
        ┌───────▼────────┐            ┌────────▼─────────┐
        │  F1 자산케어 RAG │            │  F2 사기방어 FDS  │
        │  (/chat)        │            │  (/fds)          │
        └───────┬────────┘            └────────┬─────────┘
   ┌────────────┼────────────┐                 │ 2층 평가
   ▼            ▼            ▼                 ▼
 공식문서    BM25+BGE-M3   EXAONE 생성      룰(1층)+베이스라인(2층)
 7종 색인  → RRF 검색    → 근거인용 답변   → 가중 위험점수 → 신호등🟢🟡🔴
                │                                 │
                ▼ 가드레일(근거 미달→상담사 연결)    ▼ 차단/주의/통과 + 사유
        ┌───────────────────────────────────────────────┐
        │  출력: 쉬운말 답변+출처 / 위험 신호등+권고 액션   │
        └───────────────────────────────────────────────┘
```

전 과정은 **로컬(온프레미스)**에서 동작한다. 외부 API·결제·클라우드 추론 없음. LLM은 Ollama로 띄운 **EXAONE-3.5-2.4B**, 임베딩은 **bge-m3**.

## 2. F1 자산 케어 — 하이브리드 RAG (`rag.py`)

> "주택연금 받으면 기초연금도 같이 받아요?" 같은 질문에 **근거를 찾아 그 근거 안에서만** 쉬운 말로 답하고, 모르면 지어내지 않는다.

### 2.1 데이터: 무엇을 가져오나
- **출처**: `backend/docs/` 의 마크다운 7종 — 주택연금·기초연금·예금자보호·정기예금/적금·보이스피싱·의료비 + **전북은행 시니어 예금(JB 골든에이지·JB 리치 100 — 실제 공개 상품, jbbank.co.kr)**.
- **성격**: 전부 **공개 공식 기관 원문 기반**으로 작성하고 각 문서 하단에 **인용 URL**을 명시(한국주택금융공사·보건복지부·예금보험공사·금융위·은행연합회·국민건강보험공단). 개인정보 0.
- 확장 시 JB 공개 상품설명서·약관 PDF를 같은 폴더에 넣으면 동일 파이프라인으로 색인된다.

### 2.2 가공 단계 (6단계)

**[1단계] 수집·청킹** — `_split_docs()`
- 각 `.md`를 읽어 **제목**(첫 `#` 헤딩)과 **섹션 헤딩**을 추적.
- `## 출처` 이후(인용 URL 목록)는 **검색 오염 방지를 위해 임베딩에서 제외**.
- 본문을 **문단 단위**(빈 줄/헤딩 경계)로 분할. 15자 미만은 버림.
- 각 청크에 검색 맥락을 강화하기 위해 **"제목 · 섹션 — 문단"** 형태의 `embed` 텍스트를 만든다.
  - 예: `기초연금 안내 · 받는 금액 — 혼자 사는 어르신(단독가구)은 매달 최대 342,510원을 받습니다…`
- 결과: **32개 청크** (문서 7종 기준 — 전북은행 실제 공개 상품 포함).

**[2단계] 임베딩** — `_embed()` (Ollama `bge-m3`)
- 각 청크의 `embed` 텍스트를 Ollama `/api/embeddings`로 **1024차원 벡터**로 변환 후 **L2 정규화**.
- `sha1(모델명+embed텍스트)`를 키로 `.emb_cache.json`에 캐시 → 문서가 안 바뀌면 재임베딩 0(재시작 ~수초).
- 출력: 정규화된 청크 임베딩 행렬 `emb (28 × 1024)`.

**[3단계] 색인** — BM25
- 같은 청크 텍스트를 토큰화해 **BM25Okapi** 희소 색인 생성.
- 토크나이저(`_tokenize`)는 형태소 분석기 없이 동작: 정규식 `[가-힣a-z0-9]+`로 단어 추출 + **한글 2-gram** 추가 → 한국어 부분 일치율↑.

**[4단계] 검색 — 하이브리드 + RRF** — `search()`
- 질문이 들어오면:
  - **Dense(의미)**: 질문을 bge-m3로 임베딩·정규화 → `cosine = emb · q` (정규화돼 내적=코사인).
  - **Sparse(키워드)**: BM25 점수.
- 두 랭킹을 **RRF(Reciprocal Rank Fusion)**로 융합:
  `fused_i = 1/(60 + dense_rank_i) + 1/(60 + sparse_rank_i)` → 상위 **TOP_K=4** 선택.
- 동시에 **dense 최고 유사도(top)**를 가드레일 게이트용으로 반환.

**[5단계] 가드레일 게이트(생성 전)** — `answer()`
- `top < 0.55`(RELEVANCE_THRESHOLD)면 **LLM을 호출하지 않고** 즉시 안전 응답:
  *"제가 확실히 모르는 내용이에요. 가까운 영업점 상담사에게 연결해 드릴게요."*
- 임계값 0.55는 실측 보정: 관련 질문 0.68~0.80 / 범위 밖 0.45 → 0.55가 안전 경계.
- 효과: 범위 밖 질문(예: "비트코인 사도 돼요?")에서 **환각을 원천 차단**(LLM조차 안 부름, ~0.2초).

**[6단계] 생성 + 후처리** — `_chat()` (Ollama `exaone3.5:2.4b`)
- 상위 4개 근거를 `[근거 i] (제목 · 섹션) 본문` 형태로 묶어 컨텍스트 구성.
- **시스템 프롬프트 5대 규칙**: ①근거 안에서만 답 ②답할 수 없으면 정해진 폴백 문장 ③3~4문장 쉬운 말 ④금액·나이·기간은 근거 그대로 정확히 ⑤사기·송금 의심 시 멈추라 안내.
- 옵션: `temperature=0.2`(보수적), `num_predict=320`(짧고 빠르게).
- **후처리 가드레일 2**: 답변이 "상담사에게 연결" + "모르"를 포함하면 출처를 붙이지 않음(거짓 인용 방지).

### 2.3 결과: 무엇이 나오나
```json
{ "answer": "네, 주택연금을 받으시더라도 기초연금을 함께 받을 수 있습니다. … 소득으로 간주되지 않기 때문에 …",
  "sources": ["기초연금 안내", "주택연금 안내 (한국주택금융공사)"],
  "guardrail": false }
```
- 실측 예: "예금자보호 한도?" → **"2025.9.1부터 원금+이자 1억원"** (0.9초) / "비트코인?" → 가드레일 작동(0.2초).

## 3. F2 거래단 사기 방어 — 2층 FDS (`fds.py`)

> 같은 300만원이라도 **이 사람의 평소 패턴**에 따라 통과/차단이 갈린다. LLM 불필요, 응답 ~20ms.

### 3.1 데이터: 무엇을 가져오나
거래 한 건을 다음 필드로 받는다(`/fds` 입력):

| 필드 | 의미 | 출처(실서비스) |
|---|---|---|
| `amount` | 이체 금액 | 거래 이벤트 |
| `age` | 고객 연령 | 은행 고객정보 |
| `isNew` | 신규 수취인 여부 | 거래/수취인 이력 |
| `known` | 평소 거래하던 상대인지 | 거래 이력 |
| `baseAmounts` | **이 고객의 평소 이체 금액 분포**(과거 거래) | 은행 거래내역 |
| `usualHours` | 평소 거래 시간대 | 은행 거래내역 |
| `hour` | 이번 거래 시각 | 거래 이벤트 |

- 데모에서는 **합성 데이터**(개인정보 0). 실서비스에서는 **은행이 이미 보유한 거래내역**으로 `baseAmounts`·`usualHours`를 구성(외부에서 "불러오는" 것이 아님).

### 3.2 가공: 2층 평가 — `evaluate()`

**[1층] FDS 가이드라인 룰 (설명가능)**
| 조건 | 가중치 |
|---|---|
| `age≥65 && isNew && amount≥100만원` (고령·신규수취인·고액) | **+50** |
| `hour < 6` (심야 0~6시) | **+20** |

**[2층] 개인화 베이스라인 이탈 (이 고객의 '평소' 대비)**
| 조건 | 가중치 |
|---|---|
| `known=false` (평생 처음 보는 수취인) | **+25** |
| **z-score ≥ 3** : `z = (amount − mean(baseAmounts)) / std(baseAmounts)` | **+35** |
| `hour ∉ usualHours` (평소 안 쓰던 시간대) | **+15** |
- `mean` = 평소 이체액 평균, `std` = 모표준편차(0이면 1로 보정).
- **z-score가 핵심**: "이 사람 기준에서 얼마나 비정상인가"를 통계로 측정 → 같은 금액도 사람마다 결과가 다름.

**점수 합산·판정**
- `score = min(100, Σ 가중치)`
- `score ≥ 70` → **위험 🔴**(송금 멈춤) / `≥ 35` → **주의 🟡** / 그 외 → **안전 🟢**

### 3.3 결과: 무엇이 나오나 (실측)

같은 300만원, 다른 사람:

| 고객 | 입력 요지 | 판정 | 적발 사유(가중치) |
|---|---|---|---|
| **박OO** | 평소 250만·300만도 보냄, 아는 계좌 | **안전 🟢 score 0** | 이탈 신호 없음 |
| **김OO** | 평소 20만원대, 낯선 '안전계좌' | **위험 🔴 score 100** | 1층 신규수취인 고액(+50) · 2층 낯선 수취인(+25)·z=66.0 고액(+35)·시간대(+15) |

```json
{ "score": 100, "level": "위험 🔴", "cls": "danger", "action": "stop",
  "flags": [ {"layer":1,"text":"고령 계좌 → 신규 수취인에게 3,000,000원 (100만원↑)","pts":50}, … ],
  "say": "잠깐만요. 평소와 너무 다른 거래예요. 보이스피싱이 의심됩니다 …" }
```

### 3.4 3층 — 수취계좌 평판 조회 (`reputation.py`)
- `/fds` 요청의 `account`(수취계좌)를 **사기계좌 블랙리스트**와 대조(숫자만 정규화 비교).
- 일치하면 **점수·신호등을 무시하고 즉시 🔴 완전 차단**(`blacklisted=true`), 사유 "사기 신고 이력이 있는 계좌".
- 〔**데이터**〕 데모는 `fraud_accounts.json` 샘플. **무료 공개데이터엔 계좌번호가 없으므로**(개인정보), 본 구현은 **JB 은행권 사기계좌 공유 DB / 더치트(TheCheat) API**로 대체.
- 실측: 소액·아는 패턴(평소 통과 거래)이라도 **사기계좌면 score 100·차단**.

### 3.5 공공데이터 사기 통계 (`/stats`, `fetch_stats.py`)
- 우체국 금융사기계좌 정보(공공데이터포털 15021794)·경찰청·금감원 기반 **사기유형·연령·평균피해** 통계.
- `DATA_GO_KR_KEY`(환경변수, 깃 제외) 설정 후 `fetch_stats.py` 실행 시 실데이터로 갱신, 미설정 시 번들 표본 사용.

## 4. 서빙 — FastAPI (`server.py`)

- `GET /health` → `{ok, llm, embed, docs}` (상태·모델·청크 수)
- `POST /chat` → F1 RAG 답변
- `POST /fds` → F2 사기 판정
- 시작 시 **lifespan**에서 RAG 색인을 1회 빌드. **CORS 전면 허용**(데모를 어디서 열어도 호출 가능).
- LLM이 필요 없는 `/fds`·가드레일 폴백은 즉시 응답.

## 5. 데모 연동 (`demo/index.html`)

- **자산 케어 챗** → `POST /chat`. 서버가 켜져 있으면 실제 EXAONE이 답하고, **꺼져 있으면 내장 예시 답변으로 자동 폴백**(발표장 안전장치). 연결되면 칩이 "🟢 EXAONE 실시간 연결됨"으로 바뀜.
- **사기 방어 '송금 시도'(`trySend`)** → `POST /fds`. 백엔드 판정을 받아 오버레이에 표시하고, **어느 엔진이 판정했는지 배지**(`⚙ JB FDS 엔진 · Python 백엔드`)를 띄움. 꺼지면 브라우저 내장 엔진으로 폴백.
- **본부 상황판 지도** → 카카오맵(JS SDK). 차단 발생 도시 마커가 빨갛게 깜박. 카카오 미로드 시 SVG 지도로 폴백.

## 7. 실측 성능 (로컬 CPU)

| 항목 | 수치 |
|---|---|
| 색인 빌드(최초, 32청크 임베딩) | ~40초 (이후 캐시로 ~수초) |
| F1 첫 답변(EXAONE 콜드스타트, 모델 로딩) | ~50초 (1회) |
| F1 이후 답변 | **0.9 ~ 1.6초** |
| F1 가드레일(범위 밖, LLM 미호출) | **~0.2초** |
| F2 사기 판정(`/fds`) | **~15 ~ 20ms** |

> 발표 팁: 시연 직전 워밍업 질문 1개로 EXAONE을 메모리에 올려두면 첫 답변도 1~2초.

> ⚠️ **콜드스타트 ~50초는 최초 1회 모델 로딩**이며, 상시 가동 서버에선 사용자가 겪지 않는다. **GPU/실배포에선 답변이 sub-second**(2.4B는 CPU에서도 동작하므로 H100급 인프라 불필요). 무엇보다 **송금 차단은 LLM이 아니라 룰 엔진(`/fds`, ~20ms)**이 수행하므로 골든타임과 무관하다.

## 8. 실제 구현 vs 설계·확장 (정직 구분)

| 구분 | 현재 실제 구현 | 설계·확장(미구현) |
|---|---|---|
| 검색 | bge-m3 dense + BM25 + RRF (인메모리 코사인) | Qdrant 벡터DB, Cross-Encoder Reranker |
| 한국어 토큰화 | 정규식 + 한글 2-gram(경량) | kiwipiepy 형태소 분석 |
| LLM | EXAONE-3.5-2.4B 로컬 | 도메인 파인튜닝, HyperCLOVA X SEED |
| 사기탐지 | 룰 + z-score 베이스라인 | Isolation Forest → GNN(사기링), 연합학습 |
| 데이터 | 공식문서(공개) + 합성 거래 | 은행 실거래·실상품 약관, 마이데이터 |
| 채널 | 웹 데모 + 로컬 API | 은행앱 내장, ARS/SMS 선제 개입, 통신사 신호 연계 |

**요약**: 핵심 3축(**F1 하이브리드 RAG · F2 2층 FDS · 가드레일**)은 **실제로 동작·검증됨**. 나머지는 같은 파이프라인 위의 확장 설계다.

**학습 vs 검색 (중요 구분)**: 우리 시스템은 **모델을 학습(파인튜닝)하지 않는다.** RAG=검색, FDS=룰+개인 통계(z-score)라 **적은 데이터로도 동작**한다. 검색 코퍼스는 **공개 문서**라 지금도 확장 가능(법령·공공기관·**전북/광주은행 공개 약관**). 반면 **파인튜닝·GNN 등 학습형 고도화는 JB 내부 실거래 데이터가 필요**해 *확장 단계*로 분리한다.

**정직한 한계 (발표 때 먼저 인정)**:
1. **모델 라이선스** — EXAONE은 연구·비상업이라 상용 배포는 **HyperCLOVA X로 전환**(§6).
2. **마이데이터(타행)** — 실시간 송금 차단은 **자행(이체중계단)에서만**. 마이데이터는 주기적 호출이라 **통합 조회·사후 모니터링**용(실시간 차단 아님).
3. **고령자 실사용성** — 실제 고령 사용자 필드 테스트 미실시(향후 과제).
4. **정량 측정** — 환각률·오탐율(FAR)의 대규모 정식 측정 미실시(간이 평가만, §7-1).
